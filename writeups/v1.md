# Which Whisper Serving Framework Wins? A 30-Cell Benchmark

**TL;DR:** I benchmarked Hugging Face Transformers, faster-whisper, and vLLM serving Whisper-large-v3 on L4 and A100, swept across concurrencies 1, 8, 32, 64, and 128 (50 reproducible 30-second LibriSpeech clips, 3 iterations each). The fastest framework hallucinates 50-186% of words. The most accurate framework can't scale with concurrency. The cheapest cost-per-audio-hour comes from a $0.60/hr GPU, not a $1.50/hr GPU. Repo: [whisper-serving-bench](https://github.com/dyl5051/whisper-serving-bench). Methodology + raw results published under `results/published/v1_partial/`.

## The hook

Run this benchmark and you find a finding that contradicts what the GitHub stars suggest:

> **vLLM × A100 at concurrency=8: 285× faster than real-time, $0.0048 per audio-hour. WER: 113%.**

That's the cheapest, fastest cell in the entire matrix. It's also unshippable. vLLM's Whisper integration ships without the standard ASR safeguards (compression-ratio threshold, no-speech threshold, temperature fallback) that faster-whisper applies, and the model freelances when handed silence. On 30-second clips with natural pauses, it inserts hallucinated text everywhere.

This is the kind of result that benchmarks usually hide — pick a metric, build a story, ignore the contradicting axis. Here we report all of them.

## What "v1" is

v1 is a deliberate scope cut: ship a narrow benchmark publicly, iterate in the open. **30 cells** total:

- 3 frameworks: HF Transformers (the no-batching baseline), faster-whisper (CTranslate2-compiled), vLLM (continuous batching)
- 2 GPUs: NVIDIA L4 24GB (~$0.60/hr), A100 SXM4-80GB (~$1.50/hr) — both rented from RunPod
- 5 concurrencies: 1, 8, 32, 64, 128 — closed-loop async workers pulling from a shared queue
- Whisper-large-v3, fp16, English-only
- 50 fixed clips averaging 35.5s, 3 iterations per cell, full per-request JSONs

50-cell version included Ray Serve and NVIDIA Triton; both were deliberately deferred to v1.1+ once the headline story stood up without them. The interesting Triton story requires TensorRT-LLM compilation; vanilla Triton mostly duplicates the HF baseline.

## How we measure (the boring but load-bearing part)

Every measurement choice is logged in `docs/METHODOLOGY.md`. The non-obvious ones:

- **WER is computed after Whisper's English text normalizer.** Frameworks emit different default punctuation/casing — unnormalized WER comparisons are not meaningful.
- **Closed-loop concurrency, not Poisson arrivals.** N async workers each wait for their previous response before submitting the next. Models "active concurrent users."
- **All cells run 10 sequential warmup requests + 3 iterations of the eval set timed.** Median across iterations for headline; stddev reported.
- **Background `nvidia-smi` sampler at 1 Hz.** GPU utilization tells you whether the framework actually saturated the GPU or you measured a CPU bottleneck.
- **Failures are findings.** OOM at high concurrency, framework refusing a config — recorded as `degraded`/`error`/`unsupported` in the results JSON, never silently dropped.

The harness produces a versioned per-cell JSON with full provenance: git SHA, Docker tag, GPU + driver version, hostname, every per-request timestamp.

## The four headline findings

### 1. The fastest framework is unusable

vLLM's continuous batching extracts 16-26× the throughput of faster-whisper at the same concurrency. Cost-per-audio-hour at the cheapest cell:

| framework | best cell | $/audio-hour | WER |
|---|---|---|---|
| **vLLM** | A100 c=32 | **$0.0042** | 97.5% |
| **faster-whisper** | A100 c=8 | $0.142 | 5.4% |
| **HF baseline** | L4 c=8 | $0.0265 | 1.44% |

vLLM is **30× cheaper than faster-whisper** at peak. And **180× cheaper than HF baseline**. But the WER:

| GPU | c=1 | c=8 | c=32 | c=64 | c=128 |
|---|---|---|---|---|---|
| L4 | 100.6% | 186.2% | 107.0% | 56.0% | (OOM) |
| A100 | 107.2% | 113.1% | 97.5% | 56.3% | 57.9% |

**Read these numbers slowly.** WER >100% means the framework outputs *more* hallucinated words than the reference contained. WER ~57% at the highest concurrencies is "merely" wrong on every other word.

This is a vLLM Whisper-integration issue, not a vLLM-the-engine issue. The continuous-batching engine works as advertised. The Whisper wrapper around it skips the safeguards that the openai-whisper reference implementation (and faster-whisper, which descends from it) apply by default. **Until vLLM upstream fixes this, vLLM × Whisper is a benchmark trap, not a deployment.**

### 2. HF baseline is the quality champion — and can't scale

Hugging Face Transformers' direct `model.generate()` produces **1.44% WER, identical across every cell on every GPU.** Lower than faster-whisper's 3-5%; in the same neighborhood as Whisper-large-v3's published test-clean numbers. Best transcription quality of the three frameworks.

The throughput cost: HF doesn't scale with concurrency. RTF (real-time factor — wall-clock seconds per second of audio, lower=faster) is **flat from c=1 to c=128** at ~0.044 on L4:

| L4 RTF | c=1 | c=8 | c=32 | c=64 | c=128 |
|---|---|---|---|---|---|
| HF | 0.0446 | 0.0442 | 0.0442 | 0.0442 | 0.0456 |

Why? Two structural reasons:

- **No batching.** Each `generate()` call processes one audio file. Multiple concurrent requests don't share a forward pass.
- **`model.generate()` is not thread-safe.** PyTorch's generation loop has internal state that corrupts under concurrent access — first concurrent call triggers a CUDA device-side assert and the GPU is wedged. Realistic naive serving is therefore **lock-serialized**: N concurrent requests queue through one model on one GPU.

The lock makes the gap measurable. p95 latency scales linearly:

| L4 p95 (s) | c=1 | c=8 | c=32 | c=64 | c=128 |
|---|---|---|---|---|---|
| HF | 1.96 | 13.75 | 52.09 | 102.11 | 209.12 |

Submit one request, get it back in ~2s. Submit 128 concurrent requests, the unlucky tail waits in line for ~3.5 minutes. That serialization is *exactly* the gap that batched (vLLM continuous batching) and replicated (Ray Serve) frameworks fill — in production. The HF baseline isn't broken; it's a faithful "what happens if you `pip install transformers` and don't think about serving."

### 3. L4 wins cost-per-audio-hour. A100 wins latency. Match the GPU to the workload.

The cheapest *correct* cell in the entire matrix:

| | $/audio-hour | WER |
|---|---|---|
| **HF × L4 c=8** | **$0.0265** | 1.44% |
| HF × A100 c=8 | $0.0997 | 1.44% |
| FW × L4 c=1 | $0.1004 | 4.0% |
| FW × A100 c=1 | $0.1424 | 4.3% |

L4 + HF baseline is **3.7× cheaper than A100 + HF baseline** with identical quality. If you're transcribing a backlog of podcasts at 3am — batch workload, latency doesn't matter — the cost-correct GPU is the cheap one.

A100 wins on per-request latency at low concurrency (when it's not contended — see caveats below). For real-time use cases (live streaming, voice agents), A100's per-request 2-3× advantage is what you're buying.

The decision rule:

- **Batch + cost-sensitive → L4 + HF or faster-whisper.** Cheapest. Quality on par with industry standards.
- **Real-time + latency-sensitive → A100 + faster-whisper or vLLM.** Faster per request. Costs 2-4× more.
- **Throughput at any cost, accuracy-permissive → vLLM, with the caveat that "accuracy-permissive" here means "willing to ship 50%+ word error.** Probably nobody.

### 4. OOM walls are findings

The matrix has gaps. faster-whisper × L4 stops at c=8: the 24 GB GPU runs out of memory at c=32 because each request holds full 30s context. faster-whisper × A100 stops at c=32 for the same reason on the larger 80 GB card. vLLM × L4 hits OOM at c=128.

Those `—` cells in the matrix aren't omissions — they're how each framework + GPU pair structurally caps. Useful infrastructure-decision input: if you need c=128 on a single L4, you're choosing between HF (works but doesn't scale) and finding a different stack entirely.

## Caveats (because nothing is clean)

**HF × A100 wall-clock numbers carry ~1.5-2.3× host-contention bias.** RunPod's A100 inventory in US-MO-1 was running with noisy neighbors during data collection — load avg 8-26 on the host kernel, `nvidia-smi` queries timing out >2s under contention. GPU util on HF × A100 cells came back at 20-32% (vs L4's ~92%) — clear signal that the bottleneck was *CPU* (audio decode, log-mel feature extraction, generate-loop Python overhead), not GPU. On a healthy A100 host, expect HF × A100 to beat L4 on per-request latency. The within-sweep relationships (WER stability, RTF flatness, p95 linear scaling) are correct and reproducible. Absolute wall-clock numbers should be read as upper bounds.

It took four pod redeploys in US-MO-1 to land on a host quiet enough to finish the sweep. The first three either crashed mid-run or produced wall times 5× the L4 baseline. Cloud GPU benchmarks have inherent host-quality variance; v1 ships with this caveat documented rather than re-run on a curated host. v1.0.1 may revisit if needed.

**WER on LibriSpeech-clean derived audio is artificially low** for production speech. WER here is a sanity check (does the framework break the model?), not a model-accuracy claim for noisy/accented production audio. v2 will use messier audio.

**Per-framework Docker images, not one big image.** vLLM and Triton fight over PyTorch + CUDA versions. We don't try to unify them. The base image ships only the harness + HF baseline; framework images extend it.

## Reproducing this yourself

Every cell is reproducible. Repo: [whisper-serving-bench](https://github.com/dyl5051/whisper-serving-bench).

```bash
# On a GPU box (RunPod, Lambda, your-cloud-of-choice)
git clone https://github.com/dyl5051/whisper-serving-bench
cd whisper-serving-bench

docker build -t whisper-bench:base .

# Materialize eval data (~3GB, one-time)
docker run --rm -v $(pwd)/data:/app/data whisper-bench:base \
    python scripts/prepare_data.py --eval-set librispeech_streaming_30s

# Run a single cell
docker run --rm --gpus all \
    -v $(pwd)/data:/app/data \
    -v $(pwd)/results:/app/results \
    whisper-bench:base \
    python scripts/run_cell.py configs/cells/v1_hf_l4_c8.yaml

# Or a full sweep
python scripts/run_sweep.py configs/sweeps/v1_hf_l4.yaml
```

If your reproduction differs from the published numbers by more than the standard deviation we report, [open an issue](https://github.com/dyl5051/whisper-serving-bench/issues). If it differs by a lot, I want to know — that's exactly the kind of variance the writeup tries to be honest about.
