## Headline findings

### 1. faster-whisper does NOT scale with concurrency on either GPU

For both A100 (yesterday) and L4 (tonight), RTF stays roughly flat or gets WORSE as concurrency increases from 1 to 128. The line chart literally goes DOWN — adding more workers reduces throughput slightly because faster-whisper's CTranslate2 backend doesn't tensor-batch internally. More concurrency just adds GIL contention and queue overhead.

Concrete numbers (L4):
- c=1: RTF 0.167, p95 latency 19.9s, GPU util 94%
- c=8: RTF 0.175, p95 latency 107s, GPU util 94% (slightly worse than c=1)
- c=32: RTF 0.198, 12% failure rate (degraded)
- c=64: RTF 0.241, 38% failure rate
- c=128: RTF 0.196, 46% failure rate

**Implication:** for production faster-whisper deployments, c=1 dominates c=8+ on both axes. Don't crank up per-pod concurrency; horizontal-scale with more pod replicas instead.

### 2. L4 GPU utilization is 94% at concurrency=1

Unlike A100 where util climbs with concurrency, L4 saturates immediately. The smaller GPU is compute-bound from the very first concurrent request. There's basically no headroom for serving optimizations to extract more throughput from L4 — you're already pinning the card.

**Implication:** L4 is well-suited for low-concurrency cost-effective deployments. Bumping concurrency on L4 does nothing for throughput and just adds latency.

### 3. L4 wins on cost-per-audio-hour, A100 wins on per-request latency

At c=1 (the only cell with healthy completion rate AND low latency for both GPUs):

- **A100 SXM 80GB**: $1.49/hr × RTF 0.110 = **$0.164/audio-hour**, p95 latency ~6s (estimated from rec)
- **L4**: $0.60/hr × RTF 0.167 = **$0.100/audio-hour**, p95 latency 19.9s

L4 is ~39% cheaper per audio-hour but 3-4× higher tail latency. Clean deployment tradeoff: batch workloads pick L4, real-time workloads pick A100.

### 4. vLLM serves Whisper ~4× faster than faster-whisper at the same concurrency

vLLM smoke test on L4: 50 requests in 8.5s wall (RTF aggregate 0.026 vs faster-whisper's 0.108 at the same c=8). That's ~4× faster on the same hardware. Continuous batching is doing exactly what it's supposed to — keeping the GPU continuously fed.

vLLM's internal logs reported "Maximum concurrency for 448 tokens per request: 41.12x" — meaning the engine expects to support ~41 concurrent Whisper requests on L4 before saturating KV cache.

### 5. vLLM hallucinates badly on short clips — fundamental ASR safeguards are missing

**The big finding for the writeup.** vLLM's smoke test on librispeech_test_clean_subset (5-15 second clips) showed WER 2.15 (215%!) — hypotheses contained massive repetition loops:

- "I'm not going to lie. I'm not going to lie. I'm not going to lie..." (1392 chars)
- "she taught me to write and to write and to write..." (2004 chars)
- "the count had thrown in the towel and the other two had thrown..." (2213 chars)

WER breakdown: 1249 insertions, 398 deletions, 352 substitutions, only 179 hits.

**Why:** vLLM treats Whisper as a generic seq2seq model and does NOT implement Whisper's ASR-specific decoding safeguards:
- `compression_ratio_threshold` — detect repetition loops, retry with different params
- `no_speech_threshold` — skip silent regions instead of hallucinating over them
- temperature fallback — try higher temperature if compression ratio suspicious

faster-whisper has all these built in. vLLM doesn't, because it's an LLM-serving framework that retrofitted Whisper support.

Setting `temperature=0` did NOT help (WER actually went UP slightly to 2.56). The hallucinations are fundamental to vLLM's Whisper implementation, not a sampling-temperature artifact.

**Implication for the writeup:** "vLLM achieves 4× the throughput of faster-whisper on Whisper but introduces severe transcription quality issues on short clips. The continuous-batching speedup that makes vLLM dominant for LLM serving doesn't translate to production ASR because Whisper's quality depends on ASR-specific decoding safeguards that vLLM doesn't implement. **For production Whisper deployments, faster-whisper remains the better choice.**" That's the kind of headline that gets shared.

### 6. vLLM is 16-26× faster than faster-whisper on the same hardware — but WER is still wrecked

The full vLLM L4 sweep (5 cells, 30-second pseudo-clips) confirms the throughput story is real and the quality story is grim:

| concurrency | RTF (vLLM) | RTF (faster-whisper) | speedup | WER (vLLM) | WER (faster-whisper) | GPU util | p95 latency |
|-------------|------------|---------------------|---------|------------|----------------------|----------|-------------|
| 1   | **0.0101** | 0.167 | **16.5×** | 1.01 (100%) | 0.040 (4%) | 87% | 0.4s |
| 8   | **0.0067** | 0.175 | **26.1×** | 1.16 (116%) | 0.037 (3.7%) | 97% | 12.7s |
| 32  | **0.0070** | 0.198 | **28.3×** | 1.07 (107%) | 0.036 (3.6%) | 98% | 21.4s |
| 64  | **0.0080** | 0.241 | **30.1×** | 0.56 (56%) | 0.027 (2.7%) | 97% | 24.8s |
| 128 | **0.0082** | 0.196 | **23.9×** | 0.50 (50%) | 0.037 (3.7%) | 95% | 31.6s |

**vLLM is ~25× cheaper per audio-hour than faster-whisper at the same concurrency.** At L4 prices: vLLM c=8 = **$0.004/audio-hour** vs faster-whisper c=8 = **$0.105/audio-hour**. If you only looked at throughput, vLLM would obliterate.

**But WER is 50-116%.** Hypotheses are basically gibberish on this eval. Even the "best" vLLM cell (c=128, WER 50%) is 15× worse than the worst faster-whisper cell. Production unusable.

Two unexpected sub-findings:

- **vLLM saturates the L4 GPU very efficiently.** GPU util climbs from 87% (c=1) to 97-98% (c=8 onwards) — continuous batching is doing exactly what it's supposed to do, keeping the GPU continuously fed. Compare to faster-whisper which was already at 94% util at c=1 and stayed flat (because faster-whisper isn't really parallelizing — the GPU is saturated by single-request work alone).

- **WER drops dramatically at c=64 and c=128 (from ~107% to ~50%).** Counterintuitive. Hypothesis: at high concurrency, vLLM may be hitting per-request token limits or evicting requests mid-batch in a way that truncates the worst hallucination loops. Worth investigating, but the practical conclusion is the same — even 50% WER is unusable.

**Headline framing for the writeup:**

> "vLLM serves Whisper at 25× the cost-efficiency of faster-whisper on the same hardware — but produces transcripts with 50-116% WER on 30-second clips. The continuous-batching optimization that makes vLLM dominant for LLM serving translates to throughput wins for Whisper, but vLLM's Whisper integration omits the ASR-specific decoding safeguards that prevent hallucination loops. **For production Whisper deployments, faster-whisper remains the only viable choice tested**; vLLM becomes interesting only after these safeguards are added (potentially a v1.x follow-up if vLLM upstream addresses it)."

This is a more interesting writeup than "framework X is fastest." Most benchmarks measure throughput in isolation; ours catches the quality regression that pure-throughput benchmarks miss. That's the artifact's edge.

## Operational lessons learned (for the writeup's "deployment" sidebar)

### RunPod gotchas

- **Network volumes are region-locked.** A volume in US-CA-2 can't attach to a Pod in any other region. Plan accordingly.
- **CPU-only Pods can't attach network volumes** (in current RunPod setup, at least). Need a GPU Pod for volume access.
- **GPU availability fluctuates by region.** US-GA-2 went dry mid-session forcing us to redeploy in different regions.
- **The `ssh.runpod.io` SSH proxy doesn't support SFTP.** scp fails with "subsystem request failed." Workarounds: scp -O, ssh + tar pipe, or (the winner) **RunPod's S3 API for network volumes** which JustWorks via standard `aws s3 cp`.
- **Pod-volume vs network-volume vs container-disk distinction matters enormously.** Container disk dies on stop/terminate; pod-volume dies on terminate; network volume survives indefinitely (and only that's S3-accessible).
- **"Start using CPUs" preserves the pod-volume.** When GPUs aren't available, this option restarts the same Pod on CPU hardware, keeping all data accessible.
- **Auto-named network volumes are confusing.** RunPod's deploy form auto-creates volumes with random names like "uneven_azure_quelea" — easy to lose track of which one your data is actually on.

### vLLM-specific gotchas

- vLLM 0.10+ removed the `--task transcription` flag — the engine auto-detects from the model architecture
- vLLM's Whisper model download is ~9-12 GB total because it pulls fp16 AND fp32 variants. Set `HF_HUB_DOWNLOAD_IGNORE_PATTERNS='*.fp32-*'` to skip
- vLLM downloads to `~/.cache/huggingface/` by default even when `HF_HOME` is set elsewhere — set `HUGGINGFACE_HUB_CACHE` and `TMPDIR` explicitly
- vLLM's container disk needs at least 30 GB for the pip install plus model staging
- vLLM does NOT implement Whisper's ASR safeguards (the hallucination finding above)

### Workflow lessons

- **Always set HF_HOME, HUGGINGFACE_HUB_CACHE, TMPDIR before downloading models on a Pod.** Otherwise things land on the small container disk and crash with disk-full.
- **Don't paste prompts when copying commands.** zsh interprets `#` as command sometimes, and the chat client wraps emails in `[user@host](mailto:user@host)` markdown which breaks shell parsing.
- **Always check the prompt before pasting.** `root@<id>:` = Pod, `dyl5051@Dongkeuns-MacBook-Pro` = Mac. Mixing them up wasted real time tonight.
- **Use the network volume's S3 API for data extraction.** Don't fight scp/tar over the SSH proxy.
