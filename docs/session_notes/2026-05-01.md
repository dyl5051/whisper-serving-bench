**Throughput is incredible.** RTF 0.0035 at c=8 means processing audio at ~285× real-time. The cheapest cell (c=32 at $0.0045/audio-hour) is **34× cheaper than faster-whisper × A100 at c=8** ($0.153/audio-hour).

But the WER pattern from L4 holds: 56-113% across the sweep. **vLLM remains unusable for production Whisper** even on the bigger GPU.

Two new observations specific to A100:

**A100 is COMPUTE-OVER-PROVISIONED for vLLM Whisper.** GPU util peaks at 65% even at c=128 (vs L4 hitting 97%). Whisper outputs are short (~100-300 tokens) and the encoder is naturally parallel; vLLM's continuous batching can't extract more from A100's 80 GB VRAM and massive compute when the underlying workload is too small to fill it. **Implication for cost optimization: if you were forced to serve vLLM Whisper in production, an A100 would be a worse cost choice than L4** because you'd pay for compute that vLLM can't use. The $0.0045/audio-hour above is impressive but L4 might land near $0.003-0.004 once vLLM is properly fed (tighter packing, smaller card).

**WER drops at c=64 and c=128 (107% → 56-58%).** Identical pattern to L4. The hypothesis (high-concurrency queue dynamics truncate worst hallucination loops) seems to hold across hardware classes. Doesn't make 56% WER usable, but interesting to characterize.

## Operational issues hit today (and resolutions)

### 1. pip dependency resolver timed out on vLLM

`pip install -e ".[faster-whisper,vllm,dev]"` hit `ResolutionTooDeep: 200000` after ~15 min of backtracking. The vLLM 0.19.0 release combined with our pyproject.toml's loose constraints created an unsolvable resolution problem.

**Fix**: install vLLM separately first with a pinned version, then install our package with the other extras:

```bash
pip install "vllm==0.14.0" --break-system-packages
pip install -e ".[faster-whisper,dev]" --break-system-packages
```

Should add `vllm==0.14.0` (or whatever version we know works) to pyproject.toml's `[vllm]` extra so future installs don't have this problem. v1.x housekeeping task.

### 2. Forgot to prepare_data on the fresh volume

First sweep attempt: 10/10 cells died with `subprocess_error` in 5-16 seconds each. Cause: eval set wasn't materialized on the new volume yet. Fast lesson for the future: **after deploying with a fresh network volume, ALWAYS run prepare_data before running anything else.**

### 3. AWS credentials don't persist across Mac terminal sessions

Re-pulling data from S3 today required re-exporting `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` because they were env vars from yesterday's session, not a persistent profile.

**Fix going forward**: run `aws configure --profile runpod` once to write credentials to `~/.aws/credentials`. Then use `--profile runpod` in commands instead of relying on env vars.
