## The two HF adapter fixes

Both committed in `892bcc1` and documented in `docs/METHODOLOGY.md` and the adapter docstring.

### Fix 1: long-form generation

**Symptom**: smoke test ran cleanly at c=1 but produced **WER 32%** with 0 insertions and ~4800 deletions out of 15381 reference words.

**Root cause**: the eval set `librispeech_streaming_30s` has clips averaging 35.5s and ranging up to 46s. Whisper's encoder accepts a fixed 30s window. The HF adapter's vanilla `processor(arr, sampling_rate=sr, return_tensors="pt")` truncated audio to 30s silently. For a 46s clip, the model only saw the first 30s, transcribed ~50 words, and the remaining ~100 words of audio got recorded as deletion errors.

**Fix**: enable transformers' long-form generation path in the direct `generate()` API (without falling back to the high-level `pipeline()` which has the torchcodec/ffmpeg fragility issue):

```python
inputs = self._processor(
    arr,
    sampling_rate=sr,
    return_tensors="pt",
    truncation=False,
    padding="longest",
    return_attention_mask=True,
)
generate_kwargs["return_timestamps"] = True
predicted_ids = self._model.generate(
    input_features,
    attention_mask=attention_mask,
    **generate_kwargs,
)
```

Three flags must be set together: `truncation=False` so features cover the full clip, `return_attention_mask=True` so the model knows the real length, `return_timestamps=True` so generation emits timestamp tokens that delimit 30s chunks. Result: WER dropped from 32% to **1.44%** on the same clips.

This is a methodology gap that affects any adapter whose eval set has clips >30s. Documented in METHODOLOGY.md as a "Long-form audio handling" section listing each adapter's chunking strategy.

### Fix 2: concurrency lock

**Symptom**: c=1 worked but c=8 onward all 150 requests failed with `RuntimeError: CUDA error: device-side assert triggered`. First request triggered a CUDA assert; everything else was downstream casualties.

**Root cause**: PyTorch `model.generate()` is **not thread-safe**. The harness uses asyncio with a default ThreadPoolExecutor — concurrent workers each run `_run()` in a different thread, all calling into the shared model. The model has internal state (cache, attention buffers, generation_config) that gets corrupted under concurrent access.

**Fix**: a `threading.Lock` around `model.generate()` in the adapter. Concurrent submissions queue through one model on one GPU. Throughput becomes identical to c=1; only latency degrades with queue depth.

**Why this is the correct baseline**, not a cop-out: the adapter docstring already commits to "no replicas, no batching." Naive single-model serving has *exactly* this property — N concurrent requests hit one model on one GPU and serialize. That's the realistic gap that production-serving frameworks (vLLM, Ray Serve, Triton) advertise filling. The lock makes the gap measurable instead of crashing.

## The noisy-host A100 incident

After L4 finished, redeployed onto an A100 SXM 80GB in US-MO-1. Bootstrap was clean: `init_pod.sh` ran in 3 min, model + eval data already cached on the volume from prior sessions, transformers pinned to 4.46.3 (the version that doesn't crash on `infer_schema`).

Smoke test (`v1_hf_a100_c1.yaml`) hit visible pathology:

| Signal | Value | What it means |
|---|---|---|
| Host load avg | 10+ | Heavy CPU contention from neighbors |
| `nvidia-smi` query | timing out >2s (normal: ~100ms) | GPU driver / PCIe bus contended |
| Our python process | 448% CPU, 31% GPU util | Starved on CPU→GPU dispatch path |
| c=1 wall time | **531.8s** | vs L4's 274.8s — A100 was 2× *slower* |
| WER | 0.0144 | Same as L4 — quality unaffected, only timing poisoned |

This is a textbook noisy-neighbor scenario. RunPod's A100 pods share physical hosts, and our host had at least one tenant hammering CPU + PCIe. HF baseline is especially vulnerable because it does substantial per-request CPU work (soundfile decode, processor log-mel extraction, generate() Python loop overhead, long-form chunking post-processing). When the host CPU is contended, per-request latency balloons.

**Decision: discard the result.** Quarantined as `s3://b8c835d0j0/data/results/v1_hf_a100_concurrent_1.POISONED-noisy-host.json`. Tomorrow we redeploy on a different physical host (RunPod doesn't pin pods to hosts on stop/start). New rule for tomorrow: **smoke-test sanity check before launching the full sweep** — if c=1 takes ≥300s on A100, redeploy.

Cost of the noisy-host detour: ~25 min × $1.50/hr = ~$0.65.

## Lessons / what to keep doing

- **Start with a smoke test before any sweep.** The noisy-host episode was caught only because the L4 numbers gave us a baseline to sanity-check against. Running the sweep first would have wasted 22 minutes plus produced numbers we'd have to throw away.
- **Quarantine, don't delete, bad data.** The `POISONED-noisy-host` rename keeps it around for the post-mortem in METHODOLOGY without polluting the analysis pipeline.
- **Ship platform infrastructure once.** Spending 10 min on `init_pod.sh` saves ~5 min × N future pod redeploys.
- **The threading-lock issue is the kind of bug that blesses your benchmark.** Hitting it forced explicit reasoning about what "naive single-model baseline" actually means — and the answer (serialize through the lock) is a better baseline than the confused implicit one we started with.
